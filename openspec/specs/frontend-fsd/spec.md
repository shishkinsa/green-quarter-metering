## Purpose

SPA `cnt_gq_web` SHALL предоставлять единый сценарий работы с домами, квартирами и показаниями через иерархию экранов (таблицы + модальные формы), реализованный по Feature-Sliced Design с типами из OpenAPI.

## Requirements

### Requirement: Стартовая страница — таблица домов

Приложение SHALL открываться на странице со списком домов. Это единственная стартовая точка входа после авторизации (маршрут `/`).

#### Scenario: Отображение таблицы домов

- **WHEN** пользователь открывает приложение
- **THEN** отображается таблица домов с колонками: **id**, наименование, адрес
- **AND** над таблицей расположена кнопка «Добавить дом»
- **AND** кнопки действий в строке («Редактировать», «Удалить») расположены горизонтально

#### Scenario: Добавление дома через модальную форму

- **WHEN** пользователь нажимает «Добавить дом»
- **THEN** открывается модальное окно с полями наименование и адрес
- **WHEN** пользователь сохраняет форму с валидными данными
- **THEN** дом создаётся через API
- **AND** модальное окно закрывается
- **AND** таблица домов обновляется

#### Scenario: Редактирование дома через модальную форму

- **WHEN** пользователь нажимает «Редактировать» в строке дома
- **THEN** открывается модальное окно с предзаполненными полями дома
- **WHEN** пользователь сохраняет изменения
- **THEN** дом обновляется через API
- **AND** таблица домов обновляется

#### Scenario: Удаление дома

- **WHEN** пользователь нажимает «Удалить» в строке дома
- **THEN** отображается подтверждение удаления
- **WHEN** пользователь подтверждает
- **THEN** дом удаляется через API
- **AND** строка исчезает из таблицы

#### Scenario: Переход к квартирам дома

- **WHEN** пользователь кликает по строке таблицы домов (не по кнопкам действий)
- **THEN** выполняется переход на страницу квартир выбранного дома (`/buildings/{buildingId}/apartments`)

### Requirement: Страница квартир дома

Страница квартир SHALL отображать квартиры выбранного дома в таблице с данными владельца и сводкой по показаниям.

#### Scenario: Отображение таблицы квартир

- **WHEN** пользователь открывает страницу квартир дома
- **THEN** отображается заголовок с наименованием дома и кнопка «Добавить квартиру»
- **AND** таблица содержит колонки: **id**, **номер квартиры**, **владелец**, **последняя дата передачи показаний**, **показание**, **статус передачи**

#### Scenario: Добавление квартиры через модальную форму

- **WHEN** пользователь нажимает «Добавить квартиру»
- **THEN** открывается модальное окно с полями номер квартиры и этаж (опционально)
- **WHEN** пользователь сохраняет форму
- **THEN** квартира создаётся через API
- **AND** таблица квартир обновляется

#### Scenario: Редактирование владельца квартиры

- **WHEN** пользователь выбирает действие редактирования владельца (кнопка в строке или отдельное действие)
- **THEN** открывается модальное окно с полями ФИО и телефон
- **WHEN** пользователь сохраняет
- **THEN** владелец сохраняется через API
- **AND** колонка «владелец» обновляется

#### Scenario: Удаление квартиры

- **WHEN** пользователь нажимает «Удалить» в строке квартиры
- **THEN** отображается подтверждение
- **WHEN** пользователь подтверждает
- **THEN** квартира удаляется через API
- **AND** строка исчезает из таблицы

#### Scenario: Переход к показаниям квартиры

- **WHEN** пользователь кликает по строке квартиры (не по кнопкам действий)
- **THEN** выполняется переход на страницу показаний квартиры (`/apartments/{apartmentId}/meter-readings`)

#### Scenario: Навигация назад к домам

- **WHEN** пользователь на странице квартир нажимает «Назад» или пункт хлебных крошек «Дома»
- **THEN** выполняется переход на стартовую страницу таблицы домов

### Requirement: Страница показаний квартиры

Страница показаний SHALL отображать историю передачи показаний по выбранной квартире.

#### Scenario: Отображение таблицы показаний

- **WHEN** пользователь открывает страницу показаний квартиры
- **THEN** отображается заголовок с номером квартиры (и домом)
- **AND** над таблицей расположена кнопка «Передать показание»
- **AND** таблица содержит колонки: **год**, **месяц**, **показание**, **дата и время передачи**
- **AND** строки отсортированы по периоду (год, месяц) по убыванию

#### Scenario: Передача показания через модальную форму

- **WHEN** пользователь нажимает «Передать показание»
- **THEN** открывается модальное окно с полями: год, месяц, значение показания
- **WHEN** пользователь сохраняет форму с валидными данными
- **THEN** показание передаётся через API
- **AND** модальное окно закрывается
- **AND** таблица показаний обновляется

#### Scenario: Ошибка валидации при передаче

- **WHEN** API возвращает ошибку валидации (например, показание меньше предыдущего)
- **THEN** пользователю показывается сообщение об ошибке
- **AND** модальное окно остаётся открытым для исправления

#### Scenario: Навигация назад к квартирам

- **WHEN** пользователь на странице показаний нажимает «Назад» или хлебные крошки до дома
- **THEN** выполняется переход на страницу квартир соответствующего дома

### Requirement: Модальные формы для всех мутаций

Все операции создания, редактирования и передачи данных SHALL выполняться через модальные окна (`ElDialog`), а не через inline-формы на странице.

#### Scenario: Единообразие модалок

- **WHEN** пользователь выполняет любое из действий: добавить дом, редактировать дом, добавить квартиру, редактировать владельца, передать показание
- **THEN** используется модальное окно с кнопками «Сохранить» / «Отмена»
- **AND** при «Отмена» модальное окно закрывается без изменений

#### Scenario: Блокировка повторной отправки

- **WHEN** модальная форма отправляет запрос к API
- **THEN** кнопка «Сохранить» блокируется до завершения запроса

### Requirement: Действия в таблице не вызывают переход

Кнопки действий в строках таблиц SHALL не инициировать переход на дочернюю страницу.

#### Scenario: Клик по кнопке «Редактировать»

- **WHEN** пользователь нажимает «Редактировать» или «Удалить» в строке
- **THEN** выполняется только соответствующее действие (модалка или подтверждение)
- **AND** переход по строке не происходит

### Requirement: Маршрутизация и layout

Приложение SHALL использовать nested routes с общим layout и хлебными крошками для иерархии дом → квартира → показания.

#### Scenario: Структура маршрутов

- **WHEN** приложение загружено
- **THEN** маршруты включают: `/` (дома), `/buildings/:buildingId/apartments`, `/apartments/:apartmentId/meter-readings`
- **AND** все страницы обёрнуты в общий `AppLayout`

#### Scenario: Хлебные крошки

- **WHEN** пользователь на странице квартир или показаний
- **THEN** отображаются хлебные крошки: «Дома» → «{название дома}» → «Квартира {номер}» (на странице показаний)

### Requirement: FSD-структура и типы из OpenAPI

Frontend-код SHALL следовать FSD: pages композируют features и entities; API-типы — из сгенерированного OpenAPI.

#### Scenario: Тонкие pages

- **WHEN** реализована любая из трёх страниц
- **THEN** page-компонент не вызывает HTTP-клиент напрямую
- **AND** мутации инкапсулированы в features (`create-building`, `update-building`, `delete-building`, `create-apartment`, `delete-apartment`, `upsert-apartment-owner`, `submit-meter-reading`)

#### Scenario: Типы entity из OpenAPI

- **WHEN** выполнен `npm run generate:api`
- **THEN** entity-слайсы `building`, `apartment`, `meter-reading` используют типы из `@/shared/api/generated/openapi`

#### Scenario: Public API слайсов

- **WHEN** код импортирует другой слайс FSD
- **THEN** импорт выполняется через `index.ts` слайса, не через deep paths

### Requirement: Обработка ошибок API

UI SHALL показывать пользователю понятные сообщения при ошибках API.

#### Scenario: Ошибка ProblemDetails

- **WHEN** backend возвращает 400/404/409 с ProblemDetails
- **THEN** пользователю показывается `ElMessage.error` с текстом из ответа
- **AND** парсинг ошибки выполняется в `shared/api`, а не в каждой feature
