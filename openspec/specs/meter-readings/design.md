## Context

Справочник домов, квартир и владельцев (`directories`) реализован. Следующий шаг MVP — цифровая передача показаний электроэнергии: владелец квартиры вводит значение счётчика, энергетик видит статус сдачи по дому за отчётный месяц.

Ограничения: Clean Architecture, FSD (Vue 3), PostgreSQL, UUID для id. Авторизация жильца в MVP отсутствует — владелец идентифицируется выбором квартиры в UI (допущение для демо).

## Goals / Non-Goals

**Goals:**

- Сущность `MeterReading` с привязкой к `Apartment` и отчётному периоду (`PeriodYear`, `PeriodMonth`)
- `POST /apartments/{apartmentId}/meter-readings` — передача/обновление показания с валидацией монотонности
- `GET /buildings/{buildingId}/meter-readings?periodYear=&periodMonth=` — сводка «сдано / не сдано» по квартирам дома
- SPA: страница «Показания» — форма ввода и таблица статуса
- Уникальность одного показания на квартиру за период; повторная передача за месяц — upsert

**Non-Goals:**

- Расчёт оплаты и интеграция с бухгалтерией
- Фото счётчика, SMS-напоминания
- Учётные записи и вход владельца квартиры
- Привязка физических приборов учёта (отдельный change)
- Детекция аномалий «потребление выше нормы» (только монотонность value)

## Decisions

### 1. Таблица `meter_readings` с уникальным периодом

**Решение:** отдельная таблица `meter_readings` с FK на `apartments` и UNIQUE (`ApartmentId`, `PeriodYear`, `PeriodMonth`).

**Причина:** явная модель «одно показание на квартиру за месяц»; простые запросы сводки через LEFT JOIN.

**Альтернатива:** хранение в JSON на квартире — отклонена: нет индексов и валидации на уровне БД.

### 2. Upsert при повторной передаче за месяц

**Решение:** POST обновляет существующую запись за тот же период (HTTP 200), если новое значение проходит валидацию монотонности относительно **предыдущих** периодов (не относительно старого значения того же месяца при понижении внутри месяца — см. риск).

**Причина:** жилец может исправить опечатку в том же месяце без дублей.

**Альтернатива:** HTTP 409 при дубликате — отклонена: хуже UX для исправления ошибки.

### 3. Валидация «не меньше предыдущего»

**Решение:** при сохранении сравнивать `value` с максимальным `value` по квартире за все периоды **строго раньше** текущего (`PeriodYear`/`PeriodMonth`); при upsert того же месяца — также не допускать `value` меньше максимума за предыдущие периоды.

**Причина:** соответствует ожиданию Василия; счётчик не может откатываться назад.

### 4. Тип значения показания

**Решение:** `numeric(18,3)` в PostgreSQL; в API — `number` (decimal).

**Причина:** достаточная точность для бытовых счётчиков; согласовано с типичными тарифными расчётами.

### 5. Сводка по дому — один endpoint с query-параметрами

**Решение:** `GET /buildings/{buildingId}/meter-readings?periodYear=2026&periodMonth=6` возвращает join `apartments` + `owners` + LEFT JOIN `meter_readings`.

**Причина:** прямое соответствие SQL-запросу из бизнес-задания; один round-trip для таблицы.

### 6. Идентификация владельца без auth

**Решение:** на странице передачи пользователь выбирает дом → квартиру из справочника; отдельной аутентификации нет.

**Причина:** MVP; полноценный вход — отдельный change.

### 7. Паттерн реализации

**Backend:** `MeterReading` в Entities; `IMeterReadingRepository`; handlers `SubmitMeterReading`, `ListBuildingMeterReadings`; методы в `ApartmentsController` / `BuildingsController`.

**Frontend:** FSD `entities/meter-reading`, `features/submit-meter-reading`, `pages/meter-readings`.

## Risks / Trade-offs

- [Нет auth — любой может передать показание за чужую квартиру] → принято для MVP; документировать в capabilities; последующий change `owner-auth`
- [Понижение value при upsert внутри месяца] → разрешить только если новое value ≥ max за предыдущие периоды; понижение относительно уже сохранённого за тот же месяц — отклонять 400
- [Строковая сортировка номеров квартир] → как в `directories`; числовая — при необходимости позже
- [Большие дома (50+ квартир)] → один запрос без пагинации на MVP; пагинация при росте

## Migration Plan

1. Миграция EF `AddMeterReadings` + опциональный seed (несколько показаний для демо)
2. Деплой backend + frontend
3. Rollback: откат миграции при отсутствии зависимых данных от показаний
