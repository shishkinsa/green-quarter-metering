## Context

ЖК «Зелёный квартал» требует ведения справочника недвижимости перед вводом показаний электроэнергии. Пользователь (энергетик) должен видеть по выбранному дому таблицу квартир с ФИО и телефоном владельца.

Ограничения: Clean Architecture, FSD, PostgreSQL, UUID для id сущностей.

## Goals / Non-Goals

**Goals:**

- Три сущности: `Building` (дом), `Apartment` (квартира), `Owner` (владелец)
- Связи: дом 1→N квартиры; квартира 1→0..1 владелец
- Endpoint для табличного представления: `GET /buildings/{id}/apartments` с join владельца
- Страница SPA с выбором дома и таблицей квартир/владельцев
- CRUD домов и квартир; назначение/обновление владельца через PUT

**Non-Goals:**

- Несколько владельцев на одну квартиру
- История смены владельцев
- Привязка приборов учёта (отдельный change)
- Замена capability `categories`

## Decisions

### 1. Три отдельные таблицы вместо универсального справочника

**Решение:** `buildings`, `apartments`, `owners` с явными FK.

**Причина:** доменная модель «дом — квартира — владелец»; проще валидация и отображение в таблице.

### 2. Владелец — отдельная таблица с UNIQUE на `ApartmentId`

**Решение:** одна запись `owners` на квартиру; upsert через `PUT /apartments/{id}/owner`.

**Альтернатива:** поля владельца в `apartments` — отклонена: отдельная сущность по требованию.

### 3. Список для таблицы — один endpoint с join

**Решение:** `GET /buildings/{buildingId}/apartments` возвращает DTO с полями квартиры и вложенными полями владельца (`ownerFullName`, `ownerPhone`).

### 4. Уникальность номера квартиры в пределах дома

**Решение:** уникальный индекс `(BuildingId, Number)`; конфликт → HTTP 409.

### 5. Паттерн реализации

**Backend:** `Building`, `Apartment`, `Owner` в Entities; handlers по сущностям; контроллеры `BuildingsController`, `ApartmentsController`.

**Frontend:** FSD-слои `entities/building`, `entities/apartment`, `entities/owner`; страница `pages/directories` с `el-table`.

## Risks / Trade-offs

- [Квартира без владельца] → nullable поля в DTO; в UI прочерк
- [Удаление дома с квартирами] → каскад или запрет; на MVP — запрет DELETE при наличии квартир (409)
- [Сортировка номеров «12» vs «2»] → на MVP строковая сортировка; числовая — при необходимости позже

## Migration Plan

1. Миграция EF `AddBuildingsApartmentsOwners` + seed (1–2 дома, несколько квартир с владельцами)
2. Деплой backend + frontend
3. Rollback: откат миграции при отсутствии зависимых данных
