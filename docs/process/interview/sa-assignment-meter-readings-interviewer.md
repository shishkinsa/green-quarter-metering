## Ответы «Василия» (если кандидат спрашивает)

| Вопрос | Ответ |
|--------|-------|
| Сколько счётчиков на квартиру? | Один электросчётчик |
| Кто сдаёт показания? | Владелец из справочника |
| Повтор за месяц? | Заменяет предыдущее значение |
| Срок сдачи? | С 20 по 25 число; в MVP не обязательно |

## Критерии оценки

| Критерий | Вес | На что смотреть |
|----------|-----|-----------------|
| Структура мышления | 25% | Вопросы заказчику, допущения, разумный MVP |
| Требования / use cases | 20% | Happy path + ошибка, понятные формулировки |
| Модель данных (ERD) | 20% | Связь с `apartments`, ключевые поля, период |
| SQL | 20% | Корректные JOIN, LEFT JOIN для «не сдано», осмысленные поля |
| REST API | 15% | 2 согласованных эндпоинта с ERD и сценариями |

### Зелёные флаги

- Явные допущения при нехватке времени
- Отдельная таблица показаний (не поле в `apartments`)
- SQL согласован с собственным ERD
- Трассировка: story → FR/UC → API → SQL

### Красные флаги

- Нет ошибочного сценария
- ERD без связи с квартирой
- SQL не соответствует предложенной схеме или не покрывает задачу Василия
- INNER JOIN вместо LEFT JOIN — пропадают квартиры без показаний

## Эталон SQL (ориентир, не единственный верный ответ)

Допустима таблица `meter_readings` с полями `apartment_id`, `period_year`, `period_month`, `value`.

```sql
SELECT
    a."Number"   AS apartment_number,
    o."FullName" AS owner_name,
    mr."Value"   AS reading_value
FROM apartments a
INNER JOIN owners o ON o."ApartmentId" = a."Id"
LEFT JOIN meter_readings mr
    ON mr."ApartmentId" = a."Id"
   AND mr."PeriodYear" = :year
   AND mr."PeriodMonth" = :month
WHERE a."BuildingId" = :building_id
ORDER BY a."Number";
```

Квартиры без владельца в MVP можно не учитывать или показать через `LEFT JOIN owners` — обсудить на защите.
